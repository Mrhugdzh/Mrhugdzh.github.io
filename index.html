DFS和BFS

发表于 2023-03-14 分类于 算法

DFS

DFS （深度优先搜索算法）

DFS 用于求解拓扑问题，采用递归的方式进行解决

算法思想：

  1. 访问顶点 v1
  2. 依次从 v1 的未访问邻接点出发，对图进行深度优先遍历，直到图中和 v 有通路的节点都被访问到
  3. 若此时图中仍有未访问节点，那么从该点开始继续进行 DFS，直至所有节点都被访问

BFS

BFS（广度优先搜索算法）

BFS 用于解决最短路径，采用队列的方式解决，并且 Dijkstar 和 Prim 算法都是基于 BFS 算法的

算法思想：

  1. 访问顶点 v1
  2. 访问 v1 的所有邻接点
  3. 依次从邻接点出发，访问各自的邻接点，直至所有节点都被访问

单例模式

单例模式是指在整个系统生命周期内，保证一个类只能产生一个实例，以确保类的唯一性。

单例模式是为了保证线程安全（在处理一片共享内存的时候，多个实例同时处理共享内存，容易出现问题。）

  • 线程安全：在拥有共享数据的多线程并行执行的程序中，线程安全的代码会通过同步机制保证各个线程都可以都可以正常且正确的执行，不会出现数据污染等意外情况。
  • 如何保证线程安全：
    1. 给共享的资源加一把锁，保证每个资源变量每时每刻至多被一个线程占用
    2. 让线程也拥有资源，不回去共享进程中的资源，

单例模式的分类：（创建实例的时间不同）

  • 懒汉模式：系统运行中，实例并不存在，当需要实例的时候，才会去创建，这种方式要考虑线程安全
  • 饿汉模式：系统一运行，就初始化创建实例，当需要时，直接调用就可以，这种方式本身就是线程安全的。但是由于非静态对象在进行初始化的时候顺序是不一致的，所以会出现未初始化的问题。

单例类的特点：

  • 构造函数和析构函数为私有类型，目的是禁止外部构造和析构
  • 拷贝构造函数和赋值构造函数为稀有类型，目的是为了禁止外部拷贝和赋值，确保实例的唯一性
  • 类中有一个获取实例的静态方法，可以全局访问

单例模式的实现：

  1. 加锁的懒汉单例模式
  2. 饿汉单例模式

C++ 智能指针

发表于 2023-03-14 分类于 C++

原始指针出现的问题：（声明和销毁的问题，对象声明周期的问题） #原始指针的问题

  1. 它的声明不能指示所指的到底是单个对象还是一个数组
  2. 它的声明没有告诉你用完是否应该销毁它
  3. 销毁是该使用 delete 机制还是析构机制
  4. 如果使用 delete，那么是是哟经 delete，还是 delete[]
  5. 很难确定在所有执行路径上都执行了恰为一次的销毁
  6. 没办法告诉你是否成为了悬空指针，即内存不再存在所指之物，指针仍然指向他们

智能指针包裹原始指针，存在四种智能指针：

  • Std::auto_ptr
  • Std::unique_ptr
  • Std::shared_ptr
  • Std::weak_ptr

独占资源使用 std::unique_ptr

Std:: unique_ptr 完全可以取代原始指针，它体现了专有所有权语义。

  • 一个非 null 的 std:: unique_ptr 始终拥有其指向的内容（不会造成悬空指针）

  • 移动一个 std:: unique_ptr 将所有权从源指针转移到目的指针，源指针被设为 null（**移动操作）

  • 拷贝一个 std:: unique_ptr 是不被允许的（出现重复销毁的问题）
    所以 std:: unique_ptr 是一个只可移动类型的，当析构时，默认情况下，采用原始指针调用delete的方式来实现。

  • Std:: unique_ptr 可以自定义删除操作，需要定义一个 Lambda 表达式

  • 当自定义删除器时，删除器类型必须作为第二个类型实参传递给 std::unique_ptr

  • Std:: unique_ptr 可以快速的转换为 std::shared_ptr

资源共享使用 std::shared_ptr

Std:: shared_ptr 用过引用计数来确保它是否是最后一个指向某种资源的指针。所以构造函数通常递增计数器，析构函数递减计数器，如果计数器递减后发现计数器值为 0，那么他就会销毁对象资源。

引用计数器暗含着性能问题：

  • Std:: shared_ptr 大小是原始指针的两倍，既含有一个指向资源的原始指，又有一个指向资源的引用计数值的原始指针。
  • 引用计数的内存必须动态分配
  • 递增递减操作必须是原子性的

因为移动构造函数的存在，所以构造函数不总是递增计数器。老的 shared_ptr 指向 null，所以移动构造比拷贝构造快

Std:: shared_ptr 同样也可以自定义删除器，但是删除器不是智能指针的一部分

控制块的创建遵循的规则：

  • Std::make_shared：总是创建一个控制块，所以 std:: make_shared 调用时对象不存在其他控制块
  • 当从独占指针上构造出 std:: shared_ptr 时会创建控制块
  • 从原始指针构造出 std:: shared_ptr 时会创建控制块
  1. 避免传给 std:: shared_ptr 构造函数原始指针，通常代替方法是使用 std::make_shared
  2. 如果必须传递原始指针，使用 new 出来的结果，不传递指针变量

Std:: shared_ptr 不能够处理数组

当 std:: shared_ptr 可能悬空时使用 std::weak_ptr

Std:: weak_ptr 通常是从 std:: shared_ptr 上创建，并且 std:: weak_ptr 不会影响对象的引用计数。

此时如果想要通过判断std::weak_ptr是否为空，并且对其操作，那么需要一个原子操作，防止其他线程修改。有两种方式：
1. std::weak_ptr::lock，返回一个shared_ptr，如果过期，那么这个shared_ptr为空
2. 使用weak_ptr为实参构造shared_ptr，如果过期，会抛出异常

Weak_ptr 的潜在使用场景：缓存、观察者列表、打破 shared_ptr 环状结构

优先使用 make_unique 和 make_shared，而非直接使用 new

Socket

数据链路层、网络层和传输层的协议是在内核中实现的，因为操作系统需要实现一组系统调用，使得应用程序可以访问这些协议提供的服务，主要是 Socket。作用有如下两个：

  1. 用于发送和接收数据，将应用程序的数据从用户缓冲区复制到 TCP/IP 内核发送缓冲区，或者接收数据。
  2. 应用程序可以通过 Socket 修改隔层协议的某些头部信息和其他数据结构，例如修改 IP 地址。

IP 协议族 #IP协议

IP 协议的两个重点：

  • IP 头部信息
  • IP 数据报的路由和转发：发生在除目标机器之外的所有主机和路由器上

IP 协议为上层提供无状态、无连接、不可靠的服务。

  • 无状态：指 IP 通信双方不同步传输数据的状态信息，因此所有 IP 数据报的发送、传输和接收都是独立的。
  • 无连接：IP 通信的双方都不长久地维持对方的信息。
  • 不可靠：不能自动完成重传，需要自己实现数据确认、超时重传等机制达到可靠传输的目的。

IPV4

头部结构

  • 4 位版本号：IP 协议的版本，对于 IPV4 来说，这里是 4，也就是 0100
  • 4 位头部长度：IP 头部有多少个 32bit 字（4 字节），头部最大为 60 字节
  • 8 位服务类型（TOS）：包括一个 3 位的优先权字段（现已被忽略），4 位的 TOS 字段和 1 位保留字段（必须置 0）。4 位的 TOS 字段分别表示：最小时延、最大吞吐量、最高可靠性和最小费用。其中最多有 1 个可以为 1.
  • 16 位字节数：IP 数据报的长度，以字节为单位。最大长度为 $2^{16}-1$，由于 MTU 的限制，长度不会有这么大
  • 16 位标识：唯一的标识主机发送的每一个数据报，初始值由系统随机生成，之后每个加一。同一个数据报的所有分片具有相同的标识。
  • 3 为标志字段：第一位保留，第二位表示禁止分片，如果此时 IP 数据报长度超过 MTU, 则返回给主机一个 ICMP 差错报文，并丢弃该数据报。第三位表示更多分片，除数据报的最后一个分片外，其他分片都要把它置位 1
  • 13 位分片偏移：
  • 8 位生存时间（TTL）：防止数据报陷入路由循环
  • 8 为协议：用来区分上层协议，ICMP 是 1，TCP：6，UDP：17
  • 16 位头部校验和：接收端使用 CRC 算法检验 IP 数据报头部
  • 32 位源 IP 地址
  • 32 位目的 IP 地址
  • 可变长的可选信息：这部分最多是 40 字节
    • 记录路由
    • 时间戳
    • 松散源路由选择
    • 严格源路由选择

数据解读：

  • 0x4 ：IPV4
  • 0x5：IP 头部长度为 20=4*5
  • 0x10：服务类型最小时延

TCP

面向连接、字节流、可靠传输

  • 面向连接

  • 字节流
    应用程序的读写次数和 TCP 模块的 TCP 报文段的个数之间没有任何关系，这取决于 TCP 缓冲区的大小。应用程序将写的操作存放在 TCP 模块的 TCP 缓冲区，然后由 TCP 发送，接收端的读也是如此。

发送端执行的写操作次数和接收端执行的读操作次数之间没有任何数量关系，这就是字节流的概念。

  • 可靠
    发送应答机制、超时重传机制，TCP 协议还会对 IP 数据报进行重排、整理，最终交付给应用层

TCP 固定头部

  • 16 位端口号
  • 32 位序号：一次 TCP 通信过程中某一个传输方向上的字节流的每个字节的编号。例如从 A 到 B 进行传输，第一个序列号初始为 ISN，下一个字节流是 1025~2049 字节，那么下一个 TCP 报文段的序号就是 ISN+1025
  • 32 位确认号：是收到的报文段序号值+1
  • 4 位头部长度：表示该 TCP 报文段有多少个 4 字节，头部最大为 60 字节
  • 6 个标志位
    • URG：紧急指针是否有效
    • ACK：确认号是否有效
    • PSH：表示接收端应用程序必须立即从 TCP 缓冲区读走数据
    • RST：表示要求对方重新连接，称携带 RST 的报文段为复位报文段
    • SYN：表示请求建立一个连接，同比报文段
    • FIN：通知对方本端要关闭连接，结束报文段
  • 16 位窗口大小：TCP 控制流量的手段，告诉对方自己的缓冲区还有多大空余
  • 16 位校验和
  • 16 位紧急指针：一个正的偏移量

TCP 头部选项
TCP 头部的最后一个选项字段是可变长的可选信息，这部分最多包含 40 字节（头部固定长度为 20 字节），TCP 头部选项结构如下：

  • Kind=0：选项表结束字段
  • 1：空操作，一般用于将 TCP 选项的总长度填充为 4 字节的整数倍
  • 2 ：最大报文段长度选项，只出现在同步报文段，即 SYN
  • 3：窗口扩大因子选项，只出现在同步报文段
  • 4：选择性确认：利用 SACK 技术确认是否已经到达的数据段不重传
  • 5：SACK 实际工作的选项
  • 8：时间戳选项，提供了较为专区的计算通信双方之间的回路时间 RTT，为 TCP 流量控制提供重要信息

半关闭状态
TCP 是全双工的（ #全双工 [[小知识点#^efe9fc]] ），通信的一方可以发送结束报文段给对方，告诉他本端已经完成了数据的发送，但允许继续接收来自对方的数据，直到对方也发送结束报文段来关闭连接，TCP 的这种状态叫做半关闭状态。

TCP 状态转移
完整的状态转移如图：

客户端连接在收到服务器结束报文段（6）之后，并没有直接进入 CLOSED 状态，而是进入到了 TIME_WAIT 状态，在这个状态，客户端需要等待一段长为 2MSL 的时间，才能完全关闭。
TIME_WAIT 存在的原因：

  1. 可靠地终止 TCP 连接
  2. 保证让迟来的 TCP 报文段有足够的时间被识别并丢弃

复位报文段
复位报文段用于通知对方关闭连接或重新建立连接，存在以下 3 中：

  • 访问不存在的端口
  • 异常终止连接
  • 处理半打开连接

TCP 数据流
TCP 报文段所携带的应用程序数据按照长度分为两种：交互数据和成块数据
使用交互数据的包含较少的字节，对实时性要求较高，telnet ssh
成块数据通常为 TCP 报文段允许的最大数据长度，对传输效率要求高


客户单每次发送的 TCP 确认报文段长度都为 0，而服务器端却在确认报文中，增加数据，这就是延时确认，方便查看本端是否有需要传送的数据。由于客户端的输入速度要比传输的慢的多，所以确认长度为 0，而服务器端即时处理，长度为数据长度。
延时确认：为了减少发送 TCP 报文段的数量

拥塞控制

  • 慢启动
  • 拥塞避免
  • 快速重传
  • 快速恢复

拥塞控制的最终受控对象是发送端向网络一次连续性写入的数据量，成为 SWND（发送窗口）。SNWD 太小，出现较高网络延迟，太大的话，出现网络拥塞

  • 接收通告窗口 RWND
  • 拥塞窗口 CWND

确认拥塞的依据：

  • 传输超时或者 TCP 重传定时器溢出（慢启动和拥塞避免）
  • 接收到重复的 ACK（快速重传和快速恢复）

EPOLLONESHOT 事件：保证一个 socket 连接在任意时刻都只被一个线程处理。对于注册了 EPOLLNESHOT 的文件描述符，操作系统最多出发其上注册的一个可读可写或者异常时间，且只出发一次。当处理完成之后，该线程就该立即重置这个 socket 上的 EPOLLONESHOT 时间，以确保 socket 下次可读，其 EPOLLIN 事件被触发。

三组 IO 复用函数的比较

Select、poll、epoll 都可以同时监听多个文件描述符，都将等待由 timeout 参数指定的超时时间，知道一个或者多个文件描述符上有事件发生返回，返回值是就绪的文件描述符的数量，返回 0 表示没有时事件发生。

  • 事件集 ：select 的 fd_set 没有将文件描述符和事件绑定；poll 的 pollfd 把文件描述符和事件都定义在其中，但是返回的都是整个用户注册的事件集合。Epoll 在内核中维护一个内核注册表，通过一个独立的系统调用 epoll_ctl 来控制往其中添加、删除、修改事件，这样，每次 epoll_wait 调用都直接从内核事件表中取得用户注册的时间，并且调用的 events 参数仅用来返回就绪的事件
  • 最大支持文件描述符数：poll 和 epoll 分别用 nfds 和 maxevents 参数指定最多监听多少个文件描述符和事件，这两个最大的为 65535，而 select 是受限制的。
  • 工作模式：select 和 poll 都只能工作在相对低效的 LT 模式，而 epoll 可以工作在 ET 高效模式，并且还支持 EPOLLONESHOT 事件。
  • 具体实现：select 和 poll 采用的是轮训的方式，时间复杂度是 O (n)。Epoll_wait 采用的是回调的方式，时间复杂度是 O (1)

汤普森采样

在贪婪算法和信心上限算法（UCB）中，使用到的都是先验知识进行的动作选取，通过之前奖励的累计来进行动作的选择。

汤普森采样是跟后验知识进行的动作选择，不是一个具体的值，也是采取一种采样的方式。首先一开始得到的是动作的先验知识，然后在获得数据的前提下，得到后验知识，然后在后验知识中进行采样，相当于是一种采样，虽然动作概率高，但是每个动作都有用被选到的可能性。

1. 介绍

与之前的e-greedy算法和UCB算法不同，前两种算法选择动作的依据是通过之前选择的动作所获得的奖励值的均值来进行动作的选择，而汤普森采样则不是根据以前采样的平均奖励去扩展这个建立得到后面的动作，而是从已有的分布中建立一种可能的模型，然后从这个模型中进行抽样选择动作。

这种方式，不仅仅是一种对已有奖励的一种准确的评估，同时模型也会对于这个奖励有一定的信心，这种信心会随着采集数据的增多变得越来越准确。这就是贝叶斯推理。
